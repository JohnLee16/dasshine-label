# Dasshine Label 质量控制体系

## 概述

Dasshine Label 采用**三层质量控制体系**，确保标注数据准确率 > 95%：

```
┌─────────────────────────────────────────────────────────────┐
│  第一层：自动规则质检                                          │
│  - 格式校验、必填项检查、逻辑一致性                            │
│  - 实时反馈，即时拦截错误                                      │
├─────────────────────────────────────────────────────────────┤
│  第二层：交叉验证 + 黄金标准                                    │
│  - 多标注员比对，计算一致性                                    │
│  - 测试题机制，监控标注质量                                    │
├─────────────────────────────────────────────────────────────┤
│  第三层：专家审核                                              │
│  - 抽样审核、争议仲裁                                          │
│  - 质量报告、人员培训                                          │
└─────────────────────────────────────────────────────────────┘
```

---

## 核心指标

### 1. 交叉验证

当多个标注员标注**同一任务**时，计算他们之间的一致性。

#### 一致率 (Agreement Rate)
```
一致率 = 最多出现的结果数 / 总标注数
```

**阈值：**
- ≥ 80%：一致性好，直接通过
- 60%-80%：一致性中等，需审核
- < 60%：一致性差，需专家介入

#### Kappa 系数 (Cohen's Kappa / Fleiss' Kappa)
```
Kappa = (P_o - P_e) / (1 - P_e)

P_o: 观察到的一致概率
P_e: 随机一致的概率
```

**解释：**
| Kappa | 一致性 |
|-------|--------|
| > 0.8 | 几乎完全一致 |
| 0.6-0.8 | 一致性好 |
| 0.4-0.6 | 一致性中等 |
| < 0.4 | 一致性差 |

### 2. 黄金标准题

**机制：**
- 项目中插入 10% 的测试题（已知标准答案）
- 标注员不知情地标注这些题目
- 系统根据答对率评估标注员能力

**用途：**
1. 实时监控标注员质量
2. 识别不合格标注员
3. 为交叉验证提供参考

### 3. 质量评分体系

```
综合评分 = 准确率×0.5 + 一致性×0.3 + 效率×0.2
```

| 指标 | 权重 | 说明 |
|------|------|------|
| 准确率 | 50% | 基于黄金题和审核结果 |
| 一致性 | 30% | 基于交叉验证 |
| 效率 | 20% | 完成速度和工作量 |

**等级体系：**

| 等级 | 条件 | 容量上限 | 单价系数 |
|------|------|----------|----------|
| 专家 | 评分≥95% + 任务>1000 | 80 | 1.5x |
| 高级 | 评分≥90% + 任务>500 | 50 | 1.3x |
| 中级 | 评分≥85% + 任务>200 | 30 | 1.1x |
| 初级 | 评分≥80% + 任务>50 | 20 | 1.0x |
| 新手 | 默认 | 10 | 0.8x |

## API 接口

### 1. 计算交叉验证

```http
POST /api/v1/quality/cross-validation
Content-Type: application/json

{
  "task_id": 1001
}
```

**响应：**
```json
{
  "task_id": 1001,
  "annotator_count": 3,
  "agreement_rate": 0.85,
  "kappa_score": 0.72,
  "is_golden": false,
  "golden_accuracy": null
}
```

### 2. 获取标注员质量评分

```http
GET /api/v1/quality/score/{user_id}
```

**响应：**
```json
{
  "user_id": 5,
  "username": "annotator_01",
  "accuracy": 0.92,
  "consistency": 0.88,
  "efficiency": 0.75,
  "overall_score": 0.87,
  "level": "intermediate",
  "suggested_level": "senior"
}
```

### 3. 审核任务

```http
POST /api/v1/quality/review
Content-Type: application/json

{
  "task_id": 1001,
  "decision": "approved",
  "score": 95,
  "feedback": "标注准确，实体边界正确"
}
```

### 4. 插入黄金标准题

```http
POST /api/v1/quality/insert-golden
Content-Type: application/json

{
  "project_id": 1,
  "ratio": 0.1
}
```

### 5. 项目质量报告

```http
GET /api/v1/quality/report/{project_id}
```

**响应：**
```json
{
  "project_id": 1,
  "project_name": "法律文书实体识别",
  "total_tasks": 5000,
  "approved_tasks": 3500,
  "approval_rate": 70.0,
  "cross_validated": 420,
  "golden_tasks": 500,
  "quality_score": 87.5
}
```

### 6. 质量排行榜

```http
GET /api/v1/quality/leaderboard?limit=20
```

## 质检流程

```
┌─────────────┐
│  标注完成    │
└──────┬──────┘
       ↓
┌─────────────────┐
│  自动规则检查    │ → 格式/必填/逻辑
└──────┬──────────┘
       ↓
┌─────────────────┐
│  是否为黄金题？  │
└──────┬──────────┘
       ↓ 是
┌─────────────────┐
│  计算黄金题准确率 │ → 低于阈值则警告
└──────┬──────────┘
       ↓
┌─────────────────┐
│  是否多人标注？  │
└──────┬──────────┘
       ↓ 是
┌─────────────────┐
│  计算交叉验证    │ → 一致率/Kappa
└──────┬──────────┘
       ↓
┌─────────────────┐
│  一致性 ≥ 80% ? │
└──────┬──────────┘
   是 ↓          ↓ 否
┌─────────┐   ┌─────────────┐
│ 自动通过 │   │ 送专家审核   │
└─────────┘   └─────────────┘
```

## 异常处理

### 标注员质量异常

**触发条件：**
- 连续 5 道黄金题错误
- 交叉验证一致率 < 60%
- Kappa 系数 < 0.4

**处理措施：**
1. 暂停分配新任务
2. 发送质量警告
3. 强制培训（重新学习标注规范）
4. 培训后测试通过方可继续

### 数据质量异常

**触发条件：**
- 项目整体通过率 < 70%
- 多人标注一致率持续偏低

**处理措施：**
1. 暂停项目
2. 检查标注规范是否清晰
3. 组织标注员培训
4. 优化自动标注模型

## 最佳实践

### 1. 黄金题设计
- 覆盖所有实体类型
- 难度分布：简单 40%、中等 40%、困难 20%
- 定期更新，防止泄露

### 2. 交叉验证策略
- 高价值数据：3人标注
- 普通数据：2人标注
- 自动标注高置信度：1人审核

### 3. 审核优先级
1. 黄金题错误
2. 交叉验证不一致
3. 自动标注低置信度
4. 随机抽样
